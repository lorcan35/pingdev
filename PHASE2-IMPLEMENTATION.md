# PingOS Phase 2 — Master Technical Design Document
## Source: Gemini Deep Think

This is the Master Technical Design Document (TDD) for PingOS Phase 2.As a Systems Architect, my goal is to transition PingOS from a collection of clever headless scripts into a resilient, standardized, decentralized Operating System for the Web. Given your hardware (DGX Spark, 128GB RAM, ARM64) and your local LLM capabilities, we can push the boundaries of Ahead-of-Time compilation and runtime self-healing without worrying about cloud API costs or latency.Here is the deep-dive implementation plan, architected specifically for a solo developer utilizing Claude Code/Codex.1. JIT Self-Healing (The Auto-Immune System)If PingOS cannot adapt to A/B tests or DOM mutations, it will die of maintenance debt. This system intercepts CDP execution failures, freezes the job queue, queries your local LLM with the new DOM context, patches the compiled binary (SiteDefinition) on disk, and seamlessly resumes.A. ArchitectureInterceptor: @pingdev/core wraps CDP DOM calls (waitForSelector, click). On TimeoutError, it emits a HEAL_REQUEST.Job Freezing: The active BullMQ worker catches HEAL_REQUEST, calls job.extendLock(30000), and updates status to healing. The HTTP request (e.g., streaming SSE) is kept alive via heartbeat pings.DOM Condenser: The headless tab executes a utility to dump the Accessibility Tree (AXTree) or a highly-pruned DOM. It recursively strips <script>, <style>, <svg>, and empty <div> wrappers, preserving only structural hierarchy and semantic attributes (aria-*, data-testid, role, class, id).Local Inference: Sends the pruned DOM and intended action to your local LM Studio/Ollama instance (http://localhost:1234/v1/chat/completions).Hot-Patcher & Resume: The LLM returns a JSON patch. PingOS verifies it via Runtime.evaluate, overwrites the SiteDefinition.json file, and resumes the CDP action.B. Step-by-Step ImplementationInstrument CDP: In @pingdev/core/cdp.ts, create withHealing(actionFn, intentMetadata).Build DOMCondenser.ts: Use cheerio or raw CDP Accessibility.getFullAXTree. If using raw HTML, strip all attributes except the semantic ones mentioned above.Prompt Engineering: "You are PingOS JIT. The intent {intent} failed using selector {old_selector}. Review this condensed DOM: {condensed_dom}. Return ONLY a JSON object: {"selectors": ["aria...", "css..."]}."Validation Loop: Execute document.querySelectorAll(newSelector).length === 1. Retry up to 2 times if it fails or returns multiple elements.AST Patching: Persist the verified selector back to the SiteDefinition.json file on disk.C. Technical Stack: TypeScript, puppeteer-core (or bare CDP), cheerio (for fast pruning), OpenAI Node SDK (pointed to localhost), BullMQ lock management.
D. Dependencies: Abstracted SiteDefinition files (logic must be separated from selectors).
E. Estimated Complexity: 2 Weeks. The DOMCondenser is the hardest part. Sending raw HTML will blow out the context window; pruning too much causes hallucination.
F. Critical Design Decisions:Synchronous Healing: Do not fail the API request. Suspend the job, heal, and resume. Your DGX local LLM is fast enough to heal in < 5 seconds.Persistent Fixes: Always write the fix to disk immediately. Ephemeral in-memory fixes mean you pay the LLM latency penalty on every subsequent run.2. POSIX for the Web (Standardized Interfaces)Currently, PingApps are silos (/v1/chat, /v1/tools). We need standard interfaces (/dev/llm, /dev/search) so AI agents can hot-swap ChatGPT for Gemini without changing their code.A. Architecture
Every web app becomes a Virtual File System (VFS) implementing standard system calls:open(uri) -> Allocates CDP tab, injects cookies, navigates.write(fd, payload) -> Maps to standard input (filling a chat box).read(fd) -> Attaches to the hash-based completion detector (stdout stream).exec(fd) -> Triggers a state transition (clicking 'Submit').B. Step-by-Step ImplementationDefine Core Interfaces: In @pingdev/std/traits.ts, define strict standard interfaces (e.g., ILLMDriver).Refactor Existing Apps: Wrap your Gemini, AI Studio, and ChatGPT apps in adapter classes that implement this generic interface.Create VFS Gateway: Refactor your Express/Fastify API. Map POST /v1/dev/llm/gemini/write internally to the generic BullMQ queue.Standardize Errors: Map web exceptions to POSIX codes: ENOENT (Selector missing/Healer failed), EACCES (Cloudflare block), EBUSY (Single concurrency tab locked).C. Technical Stack: Fastify (for low-overhead routing), Node Streams (converting hash-chunks into readable streams), Zod (schema validation).
D. Dependencies: Foundational. Must be built first.
E. Estimated Complexity: 1.5 Weeks. Mostly refactoring and strict typing.
F. Critical Design Decisions:Capability Flags: Since AI Studio has 21 models and ChatGPT basic has 1, the POSIX standard must include a capabilities() method returning supported features (vision, tools).Concurrency Handling (EBUSY): Because you run single-concurrency per tab, open() must lock the resource via Redis. Further requests queue in BullMQ or immediately return EBUSY.3. Ping Package Manager (ppm)The distribution engine. The apt-get for web drivers, allowing developers to publish, install, and hot-swap drivers natively.A. Architecture
A CLI tool that downloads a .pingpkg tarball from a Git repository. A package contains manifest.json (metadata, auth requirements), SiteDefinition.json (selectors/state machine), and adapter.ts (POSIX mapping).B. Step-by-Step ImplementationCLI Scaffold: Create @pingdev/cli/ppm using commander (ppm install, ppm update).Define Registry: Map ppm install github:pingos-org/perplexity to GitHub releases. Use node-fetch and tar to pull and unpack into /opt/pingos/apps/.Daemon Hot-Reload: Implement chokidar in the PingOS daemon to watch /opt/pingos/apps/. When a new app is detected, dynamically import() the adapter and mount the /dev/[app] route without dropping active jobs.Dashboard Sync: Modify your React 19 dashboard to read the installed apps directory and dynamically render their health/status cards.C. Technical Stack: Node.js, commander (CLI), tar, chokidar (file watching).
D. Dependencies: POSIX for the Web (packages must conform to the standard interface).
E. Estimated Complexity: 1 Week.
F. Critical Design Decisions:Zero Dependencies: PingApps should not have their own package.json with external modules. They must rely entirely on the @pingdev/core runtime. This keeps packages incredibly lightweight (< 10KB) and secure.4 & 7. The Chrome Extension Deep Dive (The Vital Missing Link)This is the holy grail. Developers hate writing CDP scripts. The extension serves two purposes: Session Extraction (auth bypass) and Passive Driver Generation (AoT Compiler).Feature 1: Session Cookie Extraction (The Auth Bridge)Automating logins via CDP fails against Cloudflare/Kasada. PingOS bypasses this by stealing the user's already verified session.Extraction: User clicks "Sync Session" in the extension popup while on claude.ai. The Background Worker calls chrome.cookies.getAll. It injects a script to extract localStorage, sessionStorage, navigator.userAgent, and viewport dimensions.Transmission: The extension sends this payload via WebSocket to the local PingOS daemon (ws://localhost:18800/sys/auth).Injection: When POSIX open() is called, PingOS intercepts the CDP tab initialization. It uses Network.setUserAgentOverride and Network.setCookies. It navigates to a blank page on the target domain, executes localStorage.setItem via Runtime.evaluate, and then navigates to the actual app. The headless browser is instantly logged in, holding the exact trust tokens of the real human browser.Feature 2: Passive Driver Generation (Web-to-API Compiler)How to generate @pingdev/recon files without writing code. We do not want a brittle "macro recorder"—we want to infer the State Machine natively.Event Trapping: The Content Script attaches capture-phase listeners (useCapture: true) for click and keydown. It captures event.composedPath() to pierce Shadow DOMs (critical for Claude/Perplexity).Tiered Fingerprinting: Synchronously calculates the best selector hierarchy (ARIA > data-testid > innerText > robust CSS path).State Mutation Tracking: It records a structural hash of the DOM before the click. It sets a MutationObserver. Once the network hits idle and the DOM stabilizes (e.g., a streaming response finishes), it records the after hash.LLM Compilation: The extension streams [State A, Event Selector, State B] to the PingOS daemon. The daemon asks the local DGX LLM: "The user clicked an element. State transitioned from input box to loading spinner to text block. Map this to a POSIX action."Export: The compiler outputs a ready-to-use SiteDefinition.json with the inferred State Machine (Idle > Generating > Done).C. Technical Stack: React, Vite, CRXJS (for MV3), WebSockets, MutationObserver API.
E. Estimated Complexity: 3-4 Weeks. Chrome extension APIs and noise-filtering are finicky.
F. Critical Design Decisions:Never record absolute XPaths. XPaths break instantly. The extension must generate semantic tiered selectors.WebSockets over HTTP: Bypassing CORS and communicating directly with your local port 18800 daemon is vastly easier via persistent WebSockets.5. More PingApps (Expanding Driver Library)Once the foundation is laid, expanding the driver library proves the model. Target high-value, highly-guarded domains.A. Architecture: Standardized POSIX implementations of heavily used Web APIs based on generated SiteDefinitions.
B. Targets:Claude (Reasoning): Proves the Auth Bridge works against heavy bot protection.Perplexity (Search): Proves the stream/hash-based completion works on complex shadow DOMs and citation arrays.X/Twitter (Social): Proves POSIX exec works for infinite scroll and complex data scraping.E. Estimated Complexity: 1-2 days per app (exponentially faster once the extension exists).
F. Critical Design Decisions: Never attempt to automate login for these High-Value Targets (HVTs). Always use the Chrome Extension session bridge.6. The Master Build OrderTo maximize momentum, minimize refactoring, and allow Claude Code / AI to parallelize effectively, build in this exact sequence:Phase 1: POSIX for the Web (The Contract)
Why: Freeze the API contract first. Refactor Gemini, AI Studio, and ChatGPT. If you build more apps before this, you will have to rewrite them all.Phase 2: Chrome Extension - Auth Bridge (The Key)
Why: You cannot build apps for Claude/Perplexity without bypassing Cloudflare. Get cookie/localStorage syncing working via WebSockets.Phase 3: More PingApps (The Stress Test)
Why: Use the new Auth Bridge to manually build Claude and Perplexity. This proves the POSIX interface works on strict sites.Phase 4: JIT Self-Healing (The Stabilizer)
Why build this 4th? Because you need real-world failure data to prompt-engineer your DGX LLM. By building Claude/Perplexity in Step 3, they will naturally break over a few weeks. Use those broken states as Test-Driven Development (TDD) for your JIT Healer.Phase 5: Chrome Extension - Passive Compiler (The Automator)
Why: Now that the runtime is stable, standardized, and self-healing, implement the DOM recording to automate the creation of future apps from days to minutes.Phase 6: Ping Package Manager (The Distributor)
Why: Finally, wrap your stable, indestructible ecosystem in a package manager to share it with the world. (Note: Claude Code can build the ppm CLI independently while you work on other tasks).
